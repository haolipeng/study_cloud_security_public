# 智能变电站协议系列-3、MMS协议示例

• 优先级我会这样排：

  1. 先修 octet-string 被当字符串解析
     这是最明确的实质性错误，改动小、风险低、收益直接。0x89 应该输出 hex 或 base64，不应走
     String::from_utf8_lossy()。这会影响日志准确性，也避免二进制数据被悄悄改写成不可逆文本。
  2. 统一 Data CHOICE tag 表和文档注释，尤其 [15]
     这是认知层面的基础问题。代码注释说 [15] obj-id，但更严谨的说明认为 [15] 是 Data 空洞、obj-id [15] 属
     于 TypeSpecification。不先统一，后面补类型时容易继续混表。
  3. 把 failure [0] 的错误码映射成 DataAccessError 名称
     这是低风险一致性修复。Write Response 已经这么做了，Read Response 只输出数字，日志可读性和一致性都差。
  4. 明确“浅解析”语义，或递归解析 array/structure
     如果目标是检测和日志可用性，递归解析结构化值很重要，但改动面更大，因为会牵涉 MmsAccessResult 的数据模
     型、JSON 输出和测试。短期可以先把注释/字段说明改成明确的“shallow parse”；中期再做结构化递归。
  5. 补 [11] generalized-time、[13] bcd、[14] boolean-array
     这属于覆盖率增强，不是当前最紧急。真实 IEC 61850 流量里常见性通常低于 boolean/integer/float/string/
     structure，但既然是标准分支，后续应补。
  6. 增强 tag class / constructed 校验和 malformed 处理
     这是鲁棒性提升，适合和递归解析一起做。单独做也可以，但需要决定遇到非法 tag 时是丢弃、保留 unknown、还
     是产生解析错误事件。
