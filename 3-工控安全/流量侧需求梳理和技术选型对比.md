IEC61850协议的报文中有获取操作和写入操作，那么我们重点解析哪些类型的报文呢？**关注如下的事件类型**



根据Claroty的工控配置告警类型进行拆解的内容如下：

| 事件类型                 | 优先级 | 安全含义                            | MMS 对应报文                                                 |
| ------------------------ | ------ | ----------------------------------- | ------------------------------------------------------------ |
| Configuration Download   | 高     | 从工程站/客户端 → 向设备下载配置    | `ObtainFile`（推送 CID/SCD 文件）`InitiateDownloadSequence` → `DownloadSegment` → `TerminateDownloadSequence` |
| Configuration Upload     | 高     | 从设备 → 上传/读取当前配置          | `FileOpen` → `FileRead` → `FileClose`（读取 CID 文件）<br/>`InitiateUploadSequence` → `UploadSegment` → `TerminateUploadSequence` |
| DCS Configuration Change | 低     | DCS 系统级配置变更                  | `Write`（写入控制块配置：SGCB / GoCB / RCB / LCB）<br/>`DefineNamedVariableList`<br/>`DeleteNamedVariableList` |
| File System Change       | 高     | 设备文件系统被修改                  | `ObtainFile`（上传文件到设备）<br/>`FileDelete`（删除文件）<br/>`FileRename`（重命名文件） |
| Firmware Download        | 低     | 向设备刷写固件                      | `InitiateDownloadSequence` → `DownloadSegment` × N → `TerminateDownloadSequence`<br/>`ObtainFile`（推送固件文件） |
| Memory Reset             | 低     | 设备内存/存储重置                   | `InitializeJournal`（清空日志）<br/>`Reset`（重置程序）<br/>`DeleteDomain`（删除程序域） |
| Mode Change              | **高** | 设备运行模式切换（RUN/STOP/TEST等） | `Start`（启动程序 → RUN）<br/>`Stop`（停止程序 → STOP）<br/>`Kill`（终止程序）<br/>`Write`（写入模式相关 LN，如 LLN0.Mod） |
| Monitor Mode             | 低     | 进入在线监视/调试模式               | `Read` × N（批量轮询数据）<br/>`GetNameList`（枚举数据模型）<br/>`InformationReport`（订阅报告） |
| Online Edit              | 低     | 在线修改设备运行逻辑/程序           | `Stop` → `InitiateDownloadSequence` → `DownloadSegment` × N → `TerminateDownloadSequence` → `Start`<br/>或 `Write`（直接修改数据对象） |
| Settings Change          | **高** | 修改设备参数/定值                   | `Write`（写入 DO/DA 值）<br/>细分：<br/>• 保护定值: `Write PTOC/PDIS.setVal`<br/>• 定值区切换: `Write SGCB.ActSG`<br/>• 控制参数: `Write xxxCTL.ctlVal` |

优先级高的是如下几项：

- Configuration Download
- Configuration Upload
- File System Change
- Mode Change
- Settings Change

## 3.1 检测场景与 Zeek 实现机制

> **前提说明：** Zeek 对 IEC 61850 MMS/GOOSE 的解析依赖 **Spicy 自研插件**（社区插件成熟度⭐⭐级），下表中的 Zeek 事件名称为基于 Spicy 插件的自定义事件命名规范（`iec61850_mms::*` / `iec61850_goose::*`）。

| 检测场景                       | 涉及协议/服务                                                | Zeek 检测原理                                                | 检测复杂度 |
| :----------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :--------- |
| **配置下载/上传**              | MMS 文件传输服务（ObtainFile / FileOpen / FileRead / FileClose） | 跟踪 MMS 文件传输服务序列，识别 SCL（.scd/.icd/.cid）配置文件的上传或下载操作 | **高**     |
| **修改工控设备配置文件**       | MMS Write 服务 / SetDataValues（写入配置类数据对象）         | 识别 MMS Write/SetDataValues 请求中目标为配置类逻辑节点（如 `LPHD`、`LLN0`、设定组 `SGCB`），匹配关键数据对象引用路径 | 中等       |
| **控制器模式变更**             | MMS Operate 服务作用于 Mode 数据对象（`LLN0.Mod`、`LPHD1.Mod`） | 识别 MMS 控制操作（Select / Operate）的目标对象引用中包含 `.Mod`（模式）属性，提取模式值（on/off/test/blocked） | 低         |
| **关键状态更改（开关分合闸）** | MMS Operate / DirectOperate 作用于 XCBR（断路器）/ XSWI（隔离开关）；GOOSE stNum 变化 | **MMS 侧：** 识别对 XCBR/XSWI 逻辑节点的控制操作（Operate / DirectOperate），提取 `ctlVal`（true=合闸 / false=分闸）；**GOOSE 侧：** 监控 GOOSE 帧中 `stNum`（状态编号）的变化，`stNum` 递增表示状态变更 | **高**     |
| **固件更新**                   | MMS 文件传输服务序列（FileDirectory → FileOpen → FileRead×N → FileClose），目标为固件文件（.fw/.bin/.hex） | 跨事件状态机：先通过 `FileDirectory` 列目录 → `FileOpen` 打开固件文件 → 多次 `FileRead` 传输数据 → `FileClose` 关闭；通过文件名扩展名或路径特征判断为固件 | **高**     |
| **特权命令执行**               | MMS 所有控制类服务（Select / SelectWithValue / Operate / DirectOperate / Cancel）+ 定义/删除数据集（CreateDataSet / DeleteDataSet） | 控制类 MMS 服务码白名单 + 源 IP 授权白名单交叉验证；非授权主站发起控制命令立即告警 | 低         |
| **设定值修改（保护定值变更）** | MMS SetDataValues 作用于 Setting Group 数据对象（如 `PTOC1.StrVal`、`PDIF1.OpSetPct`、`PTOV1.SetVal`）；SelectActiveSG（切换激活定值区） | 识别 MMS 写操作目标为保护定值类逻辑节点（PTOC/PDIF/PTOV/PIOC 等），提取写入数值，与预定义安全范围比较；监控 `SelectActiveSG` 操作判断是否切换定值区 | **高**     |



# 三、技术选型对比Suricata vs Snort

在工控协议安全的检测场景中，从以下几个维度来对比suricata和snort的匹配度：

## 3、1 工控协议支持

| 协议          | Suricata（8.0.4）      | Snort 3                                                      |
| ------------- | ---------------------- | ------------------------------------------------------------ |
| IEC 104       | 少量支持，解析字段有限 | **❌ 不支持**                                                 |
| IEC 61850 MMS | ❌ **不支持**           | MMS inspector 相对基础，可做协议识别和部分字段检测           |
| S7comm        | ❌ **不支持**           | Snort 3 支持的是 **S7CommPlus**（V3 协议，用于 S7-1200/1500），对经典 S7comm（V1/V2，用于 S7-300/400）支持有限 |
| GOOSE         | ❌ **不支持**           | ❌ **不支持**                                                 |
| SV            | **❌ 不支持**           | **❌ 不支持**                                                 |

**结论**：对于 IEC 104，**Suricata 有原生支持而 Snort 没有**；对于 MMS/S7comm，Snort 3 有基础级别的支持，但两者都需要二次开发才能达到深度检测。GOOSE/SV 两者均为空白。



## 3、2 协议支持

| 协议       | Suricata（8.0.4） | Snort 3  | 说明                         |
| ---------- | ----------------- | -------- | ---------------------------- |
| **SSH**    | 支持              | 支持     | 均可检测暴力破解、异常版本等 |
| **Telnet** | 支持              | 支持     | 均有良好支持                 |
| **FTP**    | 支持              | 支持     | 均有良好支持                 |
| **X11**    | ❌ 不支持          | ❌ 不支持 | 两者都不支持                 |

**结论**：两者在 SSH/Telnet/FTP 上差异不大。



## 3、3 PCAP 文件留存

| 能力                    | Suricata（8.0.4）                                            | Snort 3                |
| ----------------------- | ------------------------------------------------------------ | ---------------------- |
| **全量 PCAP 存储**      | 支持，pcap-log 输出模块，支持按大小/时间轮转                 | 支持                   |
| **条件触发 PCAP**       | `pcap-log` 可配合 `conditional: alerts` 仅保存触发告警的流的 PCAP | **支持，但是配置复杂** |
| **关联告警与 PCAP**     | EVE 日志中有 `pcap_filename` 字段可直接关联                  | **需手动关联**         |
| **PCAP 文件命名与管理** | 支持自定义前缀、目录、文件大小限制                           | 基础支持               |

**结论**：**Suricata 明显胜出**，条件 PCAP 留存 + EVE 关联是生产级功能。



## 3、4 流日志与 ELK 对接

| 能力                | Suricata（8.0.4）                                            | Snort 3                                          |
| ------------------- | ------------------------------------------------------------ | ------------------------------------------------ |
| **日志格式**        | EVE JSON（统一格式，涵盖 alert/flow/protocol/fileinfo/anomaly/stats 等 20+ 事件类型） | **部分支持，仅告警日志，流日志和协议日志能力弱** |
| **Flow 日志**       | 每条流结束时生成完整 flow 记录（含字节数、包数、时长、TCP 标志位、应用协议识别） | **无独立 flow 日志输出**                         |
| **Filebeat 集成**   | Filebeat 有 **官方 Suricata 模块**（`filebeat module suricata`），开箱即用 | **无官方模块，需自定义 pipeline**                |
| **Logstash Filter** | 社区有成熟的 Logstash filter 配置                            | ❌ **社区资源少**                                 |
| **协议元数据日志**  | 每个已启用的协议都有独立日志事件（如 `event_type: iec104`）  | **有限**                                         |

**结论**：**Suricata 决定性胜出**。Snort 3 在日志输出、ELK 生态集成方面远不如 Suricata。

# 四、总体技术方案

suricata + zeek是一个完整的体系结构，能够很好的兼容到基于签名的规则引擎和基于行为的分析引擎。

第一阶段我们先在suricata中以插件的形式实现



ELK 天然擅长处理结构化/半结构化数据（JSON、文本日志），而不是原始二进制报文。

把原始数据包塞进 Elasticsearch，Kibana 里看到的只是一堆 **不可读的 base64 或二进制 blob**，没法搜索、没法聚合、没法做仪表盘。



需明确需求：

客户是既要结构化的协议分析结果，又要能下载/回溯原始 pcap 文件？

**既要实时可视化分析，又要事后取证回溯**。



客户的实际平均流量？

原始报文需要保留多久？

suricata所在的节点的大致可用空间有多少？
