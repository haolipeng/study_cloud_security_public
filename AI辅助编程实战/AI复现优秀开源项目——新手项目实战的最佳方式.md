# 项目复现任务拆解提示词

我希望通过提示词让 LLM 一步一步复现整个项目。请按以下要求帮我进行任务拆解并撰写复现文档:

## 第一步:项目分析

首先,请仔细分析该开源项目,在 `docs/project-analysis.md` 中说明:

- 项目的核心功能和目标
- 使用的技术栈和版本要求
- 整体架构设计
- 主要模块划分和依赖关系
- 项目的难点和重点

如果内容太多写不下，可以将project-analysis.md文档内容拆分到多个文件中，文件名你自行命名。

## 任务拆解要求

- 每个任务应该是一个可以在 1-3 小时内完成的独立模块
- 任务之间应有清晰的依赖关系
- 按照"环境搭建 → 基础设施 → 核心功能 → 辅助功能"的顺序组织
- 第一个任务应该是环境搭建和项目初始化

## 文档要求

- 所有文档放在 `docs` 目录中(如不存在请创建)
- 使用数字前缀命名,如: `01-environment-setup.md`, `02-database-design.md`
- 在 `docs/README.md` 中提供所有文档的索引和概述
- 每个任务文档必须包含:
  - **任务目标**: 清晰描述要实现什么
  - **前置条件**: 需要先完成哪些任务
  - **涉及的文件**: 会创建或修改哪些文件
  - **实现步骤**: 不给出代码,只描述实现思路和关键点
  - **验收标准**: 如何判断任务完成
  - **测试方法**: 见下方详细说明

## 代码要求

- 请不要在文档中提供任何示例代码(无论是实现代码还是测试代码)
- 只描述实现思路、关键逻辑和注意事项
- 可以说明"创建 xxx 文件实现 yyy 功能",但不给出具体代码

## 测试要求

对于每个任务,都必须说明如何验证完成效果:

### 单元测试
- 说明需要创建哪个测试文件
- 列出需要测试的函数/方法
- 说明需要覆盖哪些测试场景(正常流程、边界情况、错误处理)

### 集成测试
- 说明如何验证多个模块协作是否正常

### 手动测试
提供具体的测试步骤,例如:
- 使用 curl 或 Postman 测试 API
- 运行特定命令查看输出
- 检查数据库或文件系统的状态

### 预期结果
- 明确说明测试通过的标准

## 其他说明

- 如果某个任务涉及数据库变更或重要配置修改,请特别标注
- 提供可能遇到的常见问题和排查思路
- 所有文档使用中文撰写
