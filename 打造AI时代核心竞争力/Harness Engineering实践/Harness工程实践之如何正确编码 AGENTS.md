# Harness工程实践之如何正确编码 AGENTS.md

`AGENTS.md` 的价值，不是把项目知识全部塞给 agent，而是让 agent 先看到当前任务最重要的信息。

很多团队一开始会不断往 `AGENTS.md` 里追加规则：忘了跑测试，就加测试要求；改坏 SQL，就加数据库规范；踩过线上事故，就贴一段复盘。团队成员都把自己想到的规则、约束和历史教训塞进去。一个月后文件膨胀到 300 行，两个月 450 行，三个月 600 行。然后你发现，agent 的表现反而变差了。

巨型 `AGENTS.md` 常见问题：

- **浪费上下文**：agent 还要读需求、代码、测试和工具输出，入口文件过长会挤占真正有用的信息。
- **埋没重点**：安全红线、验证要求等关键约束散落在长文中，agent 不一定每次都能稳定利用。
- **优先级混乱**：硬约束、经验建议、历史背景、个人偏好混在一起，agent 难以判断取舍。
- **维护失控**：旧规则没人敢删，文件只增不减，过时内容反而误导 agent。

结论很简单：`AGENTS.md` 应该做入口，不应该做百科。

## 1. 错误写法

错误的核心，是把 `AGENTS.md` 当成“什么都能放”的容器。典型内容包括：

- 完整事故复盘。
- 单个模块的实现细节。
- 已由测试、lint、类型系统保证的重复说明。
- 个人偏好式代码风格。
- 已经过时但没人清理的历史约束。

假设项目是 Python 3.11 + FastAPI + PostgreSQL，错误版可能长这样：

```markdown
# AGENTS.md

## 2024-11-03 线上事故复盘

那天支付服务在高峰期出现超时，原因是某个 SQL 没有命中索引。排查过程如下：
1. 先看了 Grafana 指标……
2. 又检查了慢查询日志……
3. 最后发现……

## user-profile 模块技术细节

- `normalizeUser()` 会把 `users.raw_profile` 里的 JSON 转成 `UserProfileDTO`。
- `avatar` 字段历史上有三种格式：相对路径、完整 URL、对象结构。
- 用户资料缓存 key 是 `user_profile:{user_id}:v2`，TTL 是 1800 秒。
- `formatDisplayName()` 里有一个兼容老数据的判断，不要动。
- 修改这个模块时要同时检查 `profile_sync_worker.py`。

## 提交前检查

- 提交前必须跑测试。
- 提交前必须跑 lint。
- 提交前必须跑类型检查。
- CI 也会自动运行 pytest、ruff 和 mypy。

## 代码风格

- 所有代码都要写得优雅。
- 所有函数必须少于 30 行。
- 所有变量名都要尽量短。
- 不要写太多类，函数式写法更优雅。
- 注释越少越好，好的代码应该自解释。

## 历史限制

- 不要升级 Python 版本，线上只能使用 Python 3.8。
- 所有接口都要兼容旧版移动端 v1 API。
- billing 模块暂时不要动，之前改出过线上问题。
...
```

这个例子的关键问题：

- 开头没有项目概览、依赖安装、测试和验证命令。
- 事故复盘太长，应该沉淀为规则、测试或监控，而不是原文塞进入口文件。
- `user-profile` 属于局部细节，不应该让所有任务默认读取。
- 提交前检查重复，CI 已强制的内容只需要给出统一验证命令。
- “函数必须少于 30 行”“注释越少越好”属于个人偏好，容易和实际场景冲突。
- “线上只能使用 Python 3.8”和当前项目 Python 3.11 冲突，过时规则比没有规则更危险。

## 2. 正确定位

`AGENTS.md` 是 agent 开始工作的默认入口。它不负责保存所有知识，只负责告诉 agent：这个项目怎么开始、哪些事不能做、遇到不同任务该去哪里继续读。

一个合格的入口文件，优先回答四个问题：

1. 这个项目是什么？
2. 开始工作前要知道哪些基础命令？
3. 哪些规则绝不能违反？
4. 遇到某类任务时，应该继续读哪个文件？

换句话说，`AGENTS.md` 只放所有任务都需要的最低共识。更细的知识，要放到离使用场景更近的地方：API 规则放 API 文档，数据库规则放数据库文档，模块背景放模块文档，能自动检查的规则交给测试、lint、类型系统和 CI。

## 3. 内容是否该放进 AGENTS.md 的判断标准

判断一条内容该不该放进 `AGENTS.md`，不要只问“有没有用”，而要问“是否值得让 agent 每次默认读取”。可以看三个条件：

- **全局有效**：多数目录、模块、任务都适用。
- **高频触发**：agent 经常会用到，而不是偶发场景。
- **可以执行**：能明确告诉 agent 做什么，而不是只说“注意一下”。

同时满足这三点，适合放在入口文件。只满足其中一两点，就不要默认塞进 `AGENTS.md`，而是放到更合适的位置。

适合写到 `AGENTS.md` 的内容类型，可以按下面这张表判断：

| 内容类型 | 应该写什么 | 示例 |
| --- | --- | --- |
| 项目概览 | 项目类型、主要技术栈、核心运行方式 | Python 3.11 + FastAPI 后端服务，使用 PostgreSQL |
| 快速开始 | 安装、测试、lint、完整验证命令 | `make setup`、`make test`、`make check` |
| 全局硬约束 | 所有任务都不能违反的安全、提交、验证要求 | 不提交密钥；数据库查询必须参数化 |
| 工作流程 | agent 做任务时必须遵守的通用步骤 | 修改现有行为前先读测试和调用方 |
| 按需阅读索引 | 不同任务应该继续阅读的专题文档 | API 任务读 `docs/agent/api-patterns.md` |

不适合默认放进去的内容，不是没有价值，而是应该换位置：

- 事故复盘：提炼成规则、测试用例或监控检查。
- 模块细节：放到对应模块的局部文档中；能用类型、测试表达的，不要只写成说明文字。
- 质量要求：能自动化的交给 lint、类型系统、测试和 CI。
- 低频流程：放到按需阅读文档，通过入口文件索引过去。

一个实用分法是：

- 多数任务都必须知道：放进 `AGENTS.md`。
- 只有特定任务需要知道：放进专题文档，再由 `AGENTS.md` 索引过去。
- 可以被工具检查：交给测试、lint、类型系统或 CI。

规则要具体，不要写口号。最好同时说明触发条件、具体动作和原因。

例如：

```markdown
- 涉及数据库 schema 变更时，必须同步更新 migration，并在 PR 说明里写明回滚方式。原因：历史上多次出现代码和 schema 不一致导致部署失败。
```

这比“注意数据库变更”有效，因为 agent 能明确判断何时触发、要做什么、为什么重要。

## 4. 大项目中如何拆分 AGENTS.md

项目变大后，不要继续加长根目录的 `AGENTS.md`。更好的做法是分层：根目录写全局规则，子目录写局部规则，长内容放专题文档。

根目录的 `AGENTS.md` 只放所有任务都需要知道的内容，比如项目概览、基础命令、安全红线和通用验证要求。

如果某个目录有自己的技术栈、测试方式或特殊约束，可以在该目录下再放一个 `AGENTS.md`。例如：

```text
my-fastapi-app/
  AGENTS.md
  services/
    payments/
      AGENTS.md
    user-profile/
      AGENTS.md
```

越靠近当前代码的 `AGENTS.md`，规则越具体。子目录文件不需要重复根目录已经写过的内容，只补充这个目录独有的差异。

专题内容也不要塞进根文件。API 规范、数据库规则、部署流程、安全要求可以拆成独立文档，再从 `AGENTS.md` 里链接过去。这样 agent 只有在任务相关时才继续读取，不会每次都背一遍完整项目知识。

最后，`AGENTS.md` 要像代码一样维护。Agent 反复犯同一个错，再补一条具体规则；规则已经能被测试、lint 或 CI 保证，就从文件里删掉；项目结构变了，也要同步更新入口文件。

## 5. 推荐模板

下面是一个高信噪比的 `AGENTS.md` 模板：

```markdown
# AGENTS.md

## 项目概览

这是一个 Python 3.11 + FastAPI 后端服务，使用 PostgreSQL 存储数据，通过 pytest、ruff 和 mypy 做质量检查。

## 快速开始

- 安装依赖：`make setup`
- 运行测试：`make test`
- 代码检查：`make lint`
- 完整验证：`make check`

## 全局硬约束

- 不得提交密钥、token、`.env`、证书等敏感文件。
- 修改现有行为前，必须先阅读相关测试和调用方。
- 任何数据库查询都必须使用参数化查询或 ORM 安全接口。
- 任务完成前必须运行与修改范围匹配的最小验证。
- 不要重构与当前任务无关的代码。

## 按需阅读

- 新增或修改 API：阅读 `docs/agent/api-patterns.md`
- 涉及数据库变更：阅读 `docs/agent/database-rules.md`
- 编写或修改测试：阅读 `docs/agent/testing-standards.md`
- 涉及部署流程：阅读 `docs/agent/deployment-notes.md`
- 涉及认证、权限、密钥：阅读 `docs/agent/security-rules.md`
```

这个模板遵循三个原则：

- **入口要短**：Agent 一进来就能知道项目是什么、怎么验证、哪些事情不能做。
- **规则要硬**：入口文件里只保留高频、全局、不可违反的约束，而不是所有“可能有用”的建议。
- **细节要拆**：专题信息不消失，只是不默认塞进上下文。需要 API 规则时再读 API 文档，需要数据库规则时再读数据库文档。

好的 `AGENTS.md` 不让 agent 背完整个项目，而是让正确的信息在正确的时机出现。可靠的 harness，靠的不是巨型指令文件，而是高信噪比的入口和按需加载的上下文。
