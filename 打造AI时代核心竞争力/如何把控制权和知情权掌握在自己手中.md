# 如何把控制权和知情权掌握在自己手中

> 在使用 AI 工具（如 Claude Code、Codex）开发项目时，如何避免失控，保持对项目的理解和掌控

## 背景问题

使用 AI 工具从零到一实现项目时，常见的困境：

- **初期**：代码量少，能 Review，清楚文件结构和架构设计
- **中期**：AI 生成代码越来越多，逐渐记不清每个文件的作用
- **后期**：文件结构模糊，跟不上 AI 的思路，失去控制感

### 问题的本质

1. **控制权倒置**：本应是"你设计，AI 实现"，变成了"AI 设计+实现，你追着理解"
2. **认知负载过大**：同时要理解架构、文件结构、代码细节，超出了工作记忆容量
3. **缺少增量理解的机制**：代码生成太快，没有给你消化和内化的时间

## 一、控制权：你是架构师，AI 是建筑工人

### 1. 永远先规划，后实现

❌ **错误做法**：
```
"帮我实现一个用户管理系统"
→ AI 自己设计+实现，你只能被动接受
```

✅ **正确做法（五步控制法）**：

#### 第1步：要求多个方案，掌握选择权

**你的提示词**：
```
"我需要一个用户管理系统，请给我3个架构方案，每个方案说明：
1. 核心架构（单体/微服务/Serverless等）
2. 技术栈选择（语言、框架、数据库）
3. 优点和缺点
4. 适用场景
5. 大致的开发复杂度

每个方案用5-8行总结，不要给详细实现"
```

**AI 可能的回复**：
```
方案A：传统单体架构 + Express + PostgreSQL
- 优点：简单直接，易于开发和部署，适合中小型项目
- 缺点：扩展性有限，模块耦合度较高
- 适用：团队小，需求明确，快速上线
- 复杂度：低

方案B：微服务架构 + NestJS + MongoDB + Redis
- 优点：模块解耦，可独立扩展，技术栈灵活
- 缺点：架构复杂，运维成本高，开发周期长
- 适用：大型项目，需要高可用和弹性伸缩
- 复杂度：高

方案C：Serverless + AWS Lambda + DynamoDB
- 优点：无需管理服务器，按需付费，自动扩展
- 缺点：冷启动延迟，厂商锁定，调试困难
- 适用：流量不稳定，成本敏感，快速原型
- 复杂度：中
```

**你的决策点**：
- 如果3个方案都不满意，可以说："我想要方案A的简单性，但用MongoDB代替PostgreSQL"
- 如果需要更多信息，可以问："方案A和方案C在成本上差异大吗？"
- 如果满意，选择一个："我选方案A，进入下一步"

---

#### 第2步：深入理解选定方案，提出调整

**你的提示词**：
```
"我选择方案A（单体架构 + Express + PostgreSQL），但我有几个问题：
1. 为什么选PostgreSQL而不是MySQL？
2. 是否需要加入Redis做缓存？
3. 用户认证用JWT还是Session？

请针对这些问题给出建议，说明理由"
```

**AI 可能的回复**：
```
1. PostgreSQL vs MySQL：
   - PostgreSQL支持更丰富的数据类型（JSON、数组）
   - 更好的并发控制和事务处理
   - 如果只是简单CRUD，MySQL也完全够用
   建议：如果需要复杂查询和JSON字段，用PostgreSQL

2. Redis缓存：
   - 初期用户量小，不需要
   - 可以先预留接口，后期再加
   建议：先不加，等性能瓶颈出现再优化

3. JWT vs Session：
   - JWT：无状态，支持分布式，但token无法主动失效
   - Session：有状态，需要存储，但可以随时撤销
   建议：如果是单机部署，用Session更简单安全
```

**你的决策点**：
- 确认或调整技术选型
- 可以说："好，用PostgreSQL和Session，不加Redis"
- 或者说："我还是想用JWT，因为后期可能要做移动端"

---

#### 第3步：要求模块划分和文件结构

**你的提示词**：
```
"按方案A实现，但先别写代码，给我：
1. 模块划分（分几个模块，每个模块负责什么）
2. 目录结构（用树形图展示）
3. 核心文件说明（每个文件的职责，一句话）
4. 模块之间的依赖关系

用简洁的格式，不要超过30行"
```

**AI 可能的回复**：
```
【模块划分】
1. auth 模块 - 用户认证和授权
2. user 模块 - 用户信息管理
3. db 模块 - 数据库连接和操作
4. api 模块 - HTTP 路由和中间件
5. utils 模块 - 工具函数

【目录结构】
src/
├── auth/
│   ├── session.js      # Session管理
│   └── middleware.js   # 认证中间件
├── user/
│   ├── model.js        # 用户数据模型
│   ├── service.js      # 用户业务逻辑
│   └── controller.js   # 用户API控制器
├── db/
│   └── connection.js   # 数据库连接池
├── api/
│   └── routes.js       # 路由定义
├── utils/
│   └── logger.js       # 日志工具
├── config/
│   └── index.js        # 配置文件
└── app.js              # 应用入口

【依赖关系】
app.js → api/routes.js → user/controller.js → user/service.js → user/model.js → db/connection.js
                       ↓
                  auth/middleware.js → auth/session.js
```

**你的决策点**：
- 检查模块划分是否合理
- 检查文件职责是否清晰
- 可以调整："把 logger.js 移到 config/ 目录下"
- 可以简化："session.js 和 middleware.js 合并成一个文件"
- 可以要求："给我画个模块依赖图，用ASCII"

---

#### 第4步：确认或调整结构

**你的提示词**：
```
"结构基本OK，但我有几个调整：
1. 把 utils/logger.js 移到 config/logger.js
2. auth 模块里加一个 validator.js 做输入验证
3. user 模块里的 model.js 改名为 schema.js，更符合我的习惯

请更新目录结构，然后我们开始实现"
```

**AI 可能的回复**：
```
【更新后的目录结构】
src/
├── auth/
│   ├── session.js      # Session管理
│   ├── middleware.js   # 认证中间件
│   └── validator.js    # 输入验证（新增）
├── user/
│   ├── schema.js       # 用户数据模型（改名）
│   ├── service.js      # 用户业务逻辑
│   └── controller.js   # 用户API控制器
├── db/
│   └── connection.js   # 数据库连接池
├── api/
│   └── routes.js       # 路由定义
├── config/
│   ├── index.js        # 配置文件
│   └── logger.js       # 日志工具（移动）
└── app.js              # 应用入口

已更新，准备开始实现
```

**你的决策点**：
- 最后检查一遍
- 确认无误后说："好，开始实现"
- 或者继续调整："再加个 middleware/ 目录，把所有中间件放一起"

---

#### 第5步：逐模块实现，保持控制

**你的提示词**：
```
"开始实现，按以下顺序，一次只做一个模块：
1. 先实现 db/connection.js（数据库连接）
2. 再实现 user/schema.js（用户模型）
3. 然后实现 user/service.js（业务逻辑）
... 以此类推

每个模块实现完后：
- 显示完整代码
- 说明这个模块做了什么
- 说明依赖了哪些其他模块
- 等我确认后再继续下一个

现在开始第1个：db/connection.js"
```

**AI 实现第一个模块**：
```javascript
// db/connection.js
const { Pool } = require('pg');
const config = require('../config');

const pool = new Pool({
  host: config.db.host,
  port: config.db.port,
  database: config.db.name,
  user: config.db.user,
  password: config.db.password,
  max: 20, // 最大连接数
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 2000,
});

module.exports = pool;
```

**AI 的说明**：
```
【模块说明】
- 创建PostgreSQL连接池
- 从config读取数据库配置
- 导出pool对象供其他模块使用

【依赖】
- config/index.js（需要先创建配置文件）

【下一步】
需要我先创建 config/index.js 吗？还是继续 user/schema.js？
```

**你的控制点**：
- 可以说："等等，先创建 config/index.js"
- 可以说："连接池配置改一下，max改成10"
- 可以说："好，继续 user/schema.js"
- 可以说："先别继续，我要测试一下这个连接是否正常"

### 完整流程总结

```
你：要求3个方案
AI：给出方案A、B、C
你：选择方案A，提出调整
AI：解释并调整
你：要求模块划分和文件结构
AI：给出结构
你：确认或调整结构
AI：更新结构
你：开始逐模块实现
AI：实现模块1
你：确认或调整
AI：实现模块2
你：确认或调整
...
```

**关键点**：
- 每一步都是你主动发起
- 每一步都有明确的决策点
- AI 只能在你确认后才能继续
- 你随时可以暂停、调整、回退

### 2. 设置"检查点"机制

在关键节点强制暂停：

```
"实现用户认证模块，分3步：
1. 先实现基础的登录/注册功能，完成后停下
2. 我测试通过后，再实现JWT token
3. 我确认后，再实现权限控制

每步完成后等我确认再继续"
```

### 3. 限制 AI 的"自由发挥"范围

```
"只实现我要求的功能，不要：
- 添加额外的'改进'
- 重构现有代码（除非我要求）
- 添加'可能有用'的功能
- 过度设计（比如添加配置系统、插件机制等）"
```

### 4. 使用"只读模式"探索

```
"先不要写代码，只是：
- 分析现有代码结构
- 告诉我需要改哪些文件
- 说明可能的影响范围
等我确认后再动手"
```

## 二、知情权：随时知道项目状态

### 1. 建立"项目仪表盘"

定期让 AI 更新这些文档：

#### `PROJECT_STATUS.md`（项目状态）
```markdown
## 当前进度
- [x] 用户认证模块（已完成）
- [🔄] 数据库层（进行中）
- [ ] API 接口层（未开始）

## 最近变更（最新3次）
1. 2026-05-10: 添加 JWT 认证
2. 2026-05-09: 重构数据库连接
3. 2026-05-08: 初始化项目结构

## 当前问题
- 数据库连接池配置需要优化
- 缺少单元测试

## 下一步计划
- 完成数据库迁移脚本
- 添加API文档
```

#### `ARCHITECTURE.md`（架构文档）
```markdown
## 系统架构
[简单的模块图]

## 核心模块
- auth/: 用户认证（JWT方式）
- db/: 数据库层（连接池模式）
- api/: REST API 接口

## 关键设计决策
- 为什么用JWT而不是Session？
- 为什么用连接池？
```

#### `FILE_MAP.md`（文件地图）
```markdown
## 核心文件（必须了解的）
- src/auth/jwt.js - JWT 生成和验证
- src/db/pool.js - 数据库连接池
- src/api/routes.js - API 路由定义

## 辅助文件（知道存在即可）
- src/utils/logger.js - 日志工具
- src/config/db.js - 数据库配置
```

### 2. 每次改动要求"变更摘要"

```
"每次写完代码后，给我一个变更摘要：
- 改了哪些文件（用 git diff 风格）
- 为什么改
- 可能的副作用
- 如何测试"
```

```bash

2. 画出模块依赖关系图
3. 指出可能的技术债务
4. 建议下一步优化方向"
```

## 三、实战技巧

### 技巧1：**"解释驱动开发"**

```
"在写代码前，先用注释解释你要做什么：

// 步骤1: 验证用户输入
// 步骤2: 查询数据库
// 步骤3: 生成 JWT token
// 步骤4: 返回响应

然后再实现每一步"
```

这样你可以先审查逻辑，再看实现。

### 技巧2：**"测试先行"**

```
"先写测试用例，告诉我：
- 这个功能应该做什么
- 边界情况是什么
- 错误处理是什么

我确认后再实现功能"
```

### 技巧3：**"增量可见"**

```
"一次只改一个文件，改完后：
1. 显示完整的文件内容
2. 说明改了什么
3. 等我确认再继续下一个文件"
```

### 技巧4：**"关键决策记录"**

创建 `DECISIONS.md`：
```markdown
## 技术决策记录

### 2026-05-10: 选择 JWT 而不是 Session
**背景**: 需要实现用户认证
**决策**: 使用 JWT
**原因**:
- 支持分布式部署
- 无需服务器端存储
**代价**:
- Token 无法主动失效
- Token 体积较大
```

## 四、应对"计划太长"的问题

### 问题
AI 给的详细实现计划太长，没耐心读完，但太简略又心里没底。

### 解决方案：分层计划法

要求 AI 提供**三层计划**：

#### 1️⃣ **30秒版本**（必读）
```
"用3-5个要点总结这次要做什么，每个要点一句话"
```
例如：
- 添加用户认证模块（JWT方式）
- 重构数据库连接层为连接池模式
- 新增3个API端点：登录、注册、刷新token

#### 2️⃣ **3分钟版本**（选读）
```
"对每个要点展开，说明：
- 为什么这样做
- 会影响哪些现有文件
- 有什么风险或需要注意的"
```

#### 3️⃣ **完整版本**（备查）
详细的实现步骤，你可以不读，但：
- 出问题时可以回来查
- AI 实现时有明确的参考
- 可以用来做代码审查的checklist

### 实用提示词模板

#### 模板1：要求"决策摘要"
```
"只告诉我需要我做决策的部分：
- 有哪些方案选择？
- 你推荐哪个？为什么？
- 我需要确认什么？"
```

#### 模板2：要求"风险前置"
```
"先告诉我：
- 这次改动最大的风险是什么？
- 会破坏现有功能吗？
- 需要我特别注意什么？"
```

#### 模板3：要求"可视化"
```
"用简单的ASCII图或列表展示：
- 文件变更清单（新增/修改/删除）
- 模块依赖关系
- 数据流向"
```

#### 模板4：设定"计划长度限制"
```
"给我一个不超过20行的实现计划，
只包含关键步骤和决策点"
```

### 分层计划完整模板

```
"请给我一个分层计划：

【核心摘要】（必读，5行以内）
- 要做什么
- 为什么这样做
- 主要影响范围

【关键决策】（需要我确认的）
- 方案选择及推荐
- 潜在风险

【实现清单】（可选，供参考）
- 详细步骤
- 文件变更列表

我会先看前两部分，确认后你再开始实现"
```

### 更激进的做法

如果你信任 AI 的判断：
```
"对于常规实现，直接做，不用给计划。
只有在以下情况才给我计划：
- 需要架构级别的改动
- 有多个方案需要选择
- 可能破坏现有功能
- 涉及安全或性能关键部分"
```

## 五、终极原则

### 🎯 你要做的3件事

1. **定方向**：架构、技术选型、模块划分 → 你决定
2. **设边界**：什么能做、什么不能做 → 你限制
3. **看结果**：代码质量、是否符合预期 → 你验收

### 🤖 AI 做的3件事

1. **提建议**：给方案、说优缺点 → 供你选择
2. **写代码**：按你的要求实现 → 执行
3. **解释**：为什么这样做、有什么影响 → 让你理解

## 七、核心思想总结

**控制权 = 你说做什么，AI 不能自作主张**
**知情权 = 你随时知道做了什么，为什么这样做**

**方法就是：小步快跑 + 频繁确认 + 文档同步**

### 计划的目的

计划的目的不是让你了解每个细节，而是：
1. ✅ 让你知道大方向对不对
2. ✅ 让你能做关键决策
3. ✅ 让你知道风险在哪里
4. ✅ 出问题时能快速定位

**你不需要理解每一步，你需要的是控制权和知情权。**

## 八、推荐的应对策略

### 1. 重新掌控节奏
- 在每个功能模块完成后，**主动暂停**，要求 AI 生成文档
- 让 AI 解释：这个模块做了什么、为什么这样设计、与其他模块如何交互
- 只有在你理解清楚后，才继续下一个模块

### 2. 要求 AI 先规划，后实现
```
"请先给我一个详细的实现计划，包括：
- 需要创建哪些文件
- 每个文件的职责
- 模块之间的依赖关系
- 整体的架构图
等我确认后再开始实现"
```

### 3. 建立项目文档体系
定期让 AI 生成/更新：
- `ARCHITECTURE.md` - 整体架构说明
- `FILE_STRUCTURE.md` - 文件目录结构和说明
- 每个模块的 `README.md` - 模块职责和接口说明

### 4. 分层理解，而非全盘掌握
你不需要记住每个文件的每行代码，而是要清楚：
- **架构层**：系统分几个大模块，各自职责
- **接口层**：模块之间如何通信
- **关键逻辑层**：核心业务逻辑在哪里

### 5. 使用可视化工具
让 AI 帮你生成：
- 依赖关系图
- 调用流程图
- 数据流图

### 6. 定期重构和整理
当感觉混乱时，停下来：
- 让 AI 分析当前代码结构
- 识别重复或混乱的部分
- 进行重构和优化
