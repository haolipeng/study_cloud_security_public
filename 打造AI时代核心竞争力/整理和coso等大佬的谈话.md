# 一、跨平台的前端技术选型

## 1）rust + react + tauri 

**优势：**打包体积小，内存占用低，安全性高。



**劣势：**Rust学习门槛高，依赖系统WebView（windows上是WebView2，linux上是WebKitGTK），相比Electron生态还不够成熟。



## 2）Electron（Node.js + web前端）

**优势：**生态最成熟，社区资源丰富，VSCode、Slack、Discord 等大型应用都在用。开发体验好，调试方便，纯前端团队可以快速上手。



**劣势：** 打包体积大（最小也要 100MB+），内存占用高。启动速度相对较慢。



## 3）Go + Fyne

**优势：** 纯 Go 开发。无需运行时依赖。打包体积小，启动快。Go 语言本身学习曲线平缓。大名鼎鼎的tailscale就是用Fyne进行开发的。

**劣势：** UI 组件库相对简陋，视觉效果难以做得很精美。生态较小，复杂 UI 实现困难。

注意：不适合需要现代化、高颜值界面的应用。



# 二、什么人在AI时代效率最高

和coso聊天的过程中，我发现他是大厂出身，从后端开发出身，做过10年的项目管理，然后也做过架构师。

即他是技术人员、产品经理，项目经理，技术架构师的多重身份，我在想一件事，在AI时代，产品经理、项目经理、架构师这种岗位的人反而能玩转AI工具并能做出有影响力的产品，为什么这么说呢？

## 1）产品经理 和AI

产品经理的核心能力是理解用户需求、做取舍和决策。说白了就是“想清楚要干什么”。

- 用户想要什么？痛点是什么。
- 这个产品的受众是谁？
- 产品做成什么样才够用？

这些问题都需要跟人聊天，需要琢磨，需要拍板。



## 2） 架构师 和 AI

架构师的核心工作不是写代码，而是做技术决策，而且决策具有两个特点：影响范围大、难以回退。

比如选择单体架构还是分布式架构，比如选择Mysql还是MongoDB，这不仅是技术偏好问题，而关乎产品的使用场景，未来的扩展路径等。这些选择一旦做出，整个系统就会沿着这个方向演进，中途换道的成本极高。



架构决策的本质是在各种约束条件下做权衡。架构是的判断力来自于对这些约束条件的理解和平衡，而不是依赖于对某个技术的喜好和熟练度。



生活举例：

就像盖房子，AI可以帮你砌墙、刷漆，但房子该盖几层、承重墙放哪、水电怎么走，得有人先想明白。架构师就是那个画图纸的人，AI是听指挥干活的施工队。



## 3）项目经理和AI

项目经理管理的是人、时间、资源和风险。AI工具让很多执行环节加速了，但谁来决定事项的优先级、谁来处理突发状况。



大白话来描述：

**项目经理干的是"让事情往前走"**

谁先做谁后座、项目卡住了怎么办、老板突然改需求 和怎么协调——这些事需要跟人打交道、需要见机行事。

在使用AI工具的过程中，也就是管理AI的时间和资源，也是管理我们自己的时间和资源。



在AI时代，会定方向、能做决策、懂得协调的人，AI是他们的放大器；只会执行指令的人，AI是他们的竞争者。



## 4）技术经理（面面俱到的多面手）

**什么都懂一点，才能更好的指挥AI干活**

​	AI可以帮你写前端代码，但问题来了——它写的对不对、好不好、能不能用，你得能看懂才行。

​	你不需要前端写得比专职前端程序员还溜，但你得知道React和Vue的区别在哪、为什么这段代码在手机上显示会错位。

​	**有这个底子，AI给你的代码你扫一眼就知道靠不靠谱，不靠谱的话你也知道该怎么提要求让它改。**



​	AI时代，缺少的是“谁能把不同领域的事串起来的”的人。

​	**你懂后端也懂前端，你就能一个人带着AI把整个Web应用撸出来。你懂代码也懂产品，你就能自己定需求自己实现，不用等别人。你懂技术也懂业务，你就能判断AI给的方案在实际场景里能不能落地。**

​	多面手的优势不是"每个方向都比专家强"，而是"能同时驾驭多个方向的AI输出"。



# 三、如何培养上述能力呢？

**先说一个反直觉的事实：AI时代，动手做的重要性反而更高了**

很多人觉得既然AI能帮我写代码，那我是不是少写点、多学点理论就行？恰恰相反。你得先自己做过，才能知道AI做得对不对。判断力不是看书看出来的，是踩坑踩出来的。

------

**第一，逼自己做完整的项目，而不是只做局部**

以前你写后端接口，前端让别人做。现在你试着自己带着AI把前端也做了。第一次肯定很痛苦，AI给的代码你半懂不懂，但你会被迫去搞清楚那些概念。做完一个项目，你就大概知道前端是怎么回事了。不用精通，但你知道坑在哪。

多做几个完整项目，你就从"只会一段"变成"能串全程"。

------

**第二，每次AI给你代码，多问一句"为什么这样写"**

不要只看能不能跑，要追问原理。AI解释得很耐心，你不懂可以让它换个方式讲。这样做的好处是，你不只是拿到代码，还拿到了背后的知识。下次遇到类似问题，你就能自己判断了。

把AI当成一个随时可以请教的高级工程师，别只当它是代码生成器。

------

**第三，刻意让自己接触不熟悉的领域**

你是后端，就去碰碰前端。你是开发，就去了解一下产品经理怎么想问题。你写业务代码，就去看看架构层面的东西怎么设计的。

不用一开始就啃大部头，从实际问题入手。比如你想做个小工具给自己用，那就逼自己前后端都搞定。做着做着，你就变成多面手了。

------

**第四，培养"定义问题"的习惯**

遇到一个需求，先别急着问AI怎么实现。先问自己几个问题：这个需求到底在解决什么问题？有没有更简单的解法？边界条件是什么？做到什么程度算够用？

这个思考过程就是在锻炼产品经理和架构师的思维方式。时间长了，你看问题的视角就不一样了。

------

**第五，主动复盘失败的案例**

AI给的代码出了bug，别光顾着让它改。搞清楚为什么会出这个bug，下次怎么避免。是你的需求没描述清楚？还是AI对某个场景理解有偏差？还是这类问题本来就有陷阱你之前不知道？

每一次失败都是涨判断力的机会。

------

**总结一下就是：多做完整的事、多追问为什么、多碰不熟的领域、多想问题本质、多从失败里学。**
